TASK-003: Implement normalizeForTOON helper with tests
Date: 2026-05-06T00:00:00Z
================================================================================

Summary:
Replaced the placeholder doc-only internal/presentation/toon.go with a pure
helper, normalizeForTOON, that recursively rewrites Neo4j entity maps into
encoder-friendly shapes ahead of the TOONFormatter (task-004). Added focused
unit tests in internal/presentation/toon_internal_test.go covering node maps,
relationship maps, plain maps, nested entities, slices of nodes, slices of
typed maps ([]map[string]any), scalars, and immutability of the input.

Files Changed:
- internal/presentation/toon.go (rewritten: now exposes the unexported
  normalizeForTOON helper plus normalizeMap/normalizeNode/normalizeRel /
  normalizeLabels/entityPropsNormalized internals; blank import for
  toon-go retained until task-004 lands the real import)
- internal/presentation/toon_internal_test.go (created: 8 test functions
  exercising all helper code paths, lives in package presentation so it
  can call the unexported helper)
- .plans/tasks-add-toon-to-output-formats.yml (status flip → completed)

Key Decisions:
- Helper is unexported (normalizeForTOON, lower-case n) because it is an
  internal implementation detail of the upcoming TOONFormatter; tests
  live in the *_internal_test.go file per AGENTS.md convention.
- Used `any` rather than `interface{}` throughout the new file for
  consistency with newer-style Go and brevity. Existing code in this
  package mixes both — both are valid.
- Labels are coerced to []any (not []string) so the TOON encoder always
  sees a homogeneous-shape array regardless of whether the upstream
  driver provided []string or []any.
- []map[string]any inputs are explicitly handled (they don't match the
  []any case in the type switch) — some callers in the codebase produce
  this shape directly. Test TestNormalizeForTOON_SliceOfMapsTyped locks
  this behaviour down.
- The helper does not mutate its input — it always allocates a new map
  / slice. TestNormalizeForTOON_DoesNotMutateInput asserts this and
  documents the contract for downstream callers.
- All keys starting with "_" are stripped (not just _labels/_type/_id) —
  this future-proofs against new Neo4j sentinel keys (e.g. _start, _end
  on relationships) appearing in repository.convertValue output.
- Blank import of toon-go retained with an updated comment that calls
  out task-004 as the deletion point. Without it, `go mod tidy` would
  remove the dependency before task-004 needs it.

TASK-004: Implement TOONFormatter using toon-go
Date: 2026-05-06T00:00:00Z
================================================================================

Summary:
Replaced the placeholder blank import in internal/presentation/toon.go with a
real toon-go import and added the TOONFormatter type alongside the existing
normalizeForTOON helper. The formatter satisfies presentation.OutputFormatter
and serialises Tabular (via tabularToJSONSlice), *DetailData (via
detailToJSONMap), strings, and arbitrary scalars. All values are routed
through normalizeForTOON before encoding so Neo4j entity maps emit cleanly
without underscore-prefixed sentinel keys. Encoder errors are wrapped with
fmt.Errorf("encode TOON: %w", err). Added a focused external-package test
file (toon_test.go) covering the documented input shapes plus a concurrency
smoke test.

Files Changed:
- internal/presentation/toon.go (modified: replaced blank import with real
  toon import + fmt; added TOONFormatter struct and Format method;
  consolidated the file-level doc comment to cover both helper and formatter)
- internal/presentation/toon_test.go (created: 7 test functions exercising
  Tabular multi-row, empty Tabular, *DetailData, scalar string, scalar int,
  Tabular with node cells, Tabular with relationship cells, and concurrent
  use of a single TOONFormatter instance)
- .plans/tasks-add-toon-to-output-formats.yml (status flip → completed)

Key Decisions:
- Format dispatch mirrors JSONFormatter exactly (Tabular → slice of row
  maps, *DetailData → flat map, default → pass-through). This keeps the
  TOON output structurally aligned with the JSON output for the same
  inputs, which is the natural mental model for users switching between
  the two formats.
- Tests live in the external presentation_test package (toon_test.go),
  not the *_internal_test.go file — the formatter is exported so tests
  use only the exported API, matching the rest of presentation_test.go.
- The TOONFormatter is not yet registered in NewPresentationService —
  that is task-005, deliberately split so the formatter and its
  registration are reviewable independently. Acceptance criterion
  "TOONFormatter satisfies presentation.OutputFormatter and is exported"
  is verified by tests calling Format directly on a &TOONFormatter{} —
  if the interface contract were broken, the file would not compile.
- Pre-existing gofmt drift in internal/presentation/data.go (an
  unrelated alignment issue around DetailData fields) was deliberately
  not touched in this task to keep the diff scoped.
